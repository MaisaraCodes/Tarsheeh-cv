## What Was Built

### 1. Interview Question Generator Agent

**File:** `backend/agents/question_generator_agent.py`

Takes a candidate's CV analysis output (specifically `matching_skills` and `missing_skills`) along with the job profile and generates 3 to 5 tailored interview questions per candidate using OpenAI `gpt-4o`.

Two types of questions are always produced:
- **Gap-probing questions** targeting `missing_skills` — to determine whether the candidate has foundational knowledge or the aptitude to close the gap.
- **Deep-dive questions** targeting `matching_skills` — to verify the actual depth of expertise, not just keyword presence on a CV.

The agent function signature is `generate_interview_questions(job_profile, candidate_id, job_id, analysis)`. Both `candidate_id` and `job_id` are explicitly set on the returned model after generation to guarantee ID integrity regardless of what the LLM returns in the JSON.

### 2. Report Agent

**File:** `backend/agents/report_agent.py`

Takes the full ranked candidates list and the job profile and produces an executive-level hiring report using OpenAI `gpt-4o`. The agent is prompted to act as an HR Consultant and writes three structured sections:

- **Executive Summary** — how the hiring process went overall.
- **Top Candidates Recommendation** — a concrete recommendation on who to hire and why, grounded in rank and skills fit.
- **Overall Hiring Insight** — a meta-level observation about the quality of the applicant pool (e.g., whether applicants match the job description or whether the JD needs adjustment).

The tone is formal and professional, suitable for direct delivery to a hiring manager.

### 3. Pydantic Models

**`backend/models/questions.py` — `InterviewQuestions`**

| Field | Type | Description |
|---|---|---|
| `candidate_id` | str | Unique identifier for the candidate |
| `job_id` | str | The job these questions belong to |
| `questions` | List[str] | 3 to 5 tailored interview questions |

The `job_id` field was added specifically to match the `/questions/{candidate_id}` response shape in the API Contract, which requires both IDs in the response body.

**`backend/models/report.py` — `HiringReport`**

| Field | Type | Description |
|---|---|---|
| `executive_summary` | str | Executive summary of the hiring process |
| `top_candidates_recommendation` | str | Recommendation on who to hire and why |
| `overall_hiring_insight` | str | General insight on applicant pool quality |

### 4. Orchestrator — Full Pipeline

**File:** `backend/agents/graph.py`

The `AgentState` was extended with three new fields:

```python
job_id: str                                     # Passed in at invocation, threaded through to questions
interview_questions: Optional[List[Dict]]       # Accumulated per-candidate question sets
final_report: Optional[HiringReport]           # Executive report at the end of the pipeline
```

Two new node functions were added:

- `interview_node_function` — loops over all `all_cv_analyses` entries and calls the Question Generator Agent for each candidate, passing the `job_id` from state.
- `report_node_function` — calls the Report Agent with the ranking result and job profile to produce the final `HiringReport`.

The graph is now a complete 5-node linear pipeline:

```
START → intake → cv_analyzer → ranking → interview → report → END
```

Node names were deliberately set to match the agreed `stage` values in the API Contract's `/status/{job_id}` response, so the frontend polling screen will display the correct stage label for each active node without any mapping layer needed.

## API Contract Alignment

All models and agents were reviewed against `docs/API_CONTRACT.md` and confirmed compliant:
