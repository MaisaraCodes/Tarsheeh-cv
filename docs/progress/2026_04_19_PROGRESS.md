## What Was Built

### 1. Database Schema

All five tables defined in the DATABASE_SCHEMA.md were created in Supabase, along with the pgvector extension for vector similarity search.

Tables:
- `jobs` - job postings and parsed profiles from the Intake Agent
- `candidates` - CV data, scorecards, interview questions
- `documents` - knowledge base chunks (flat pattern, one row per chunk)
- `embeddings` - 1536-dimensional vectors linked to document chunks
- `job_results` - final pipeline output (ranked candidates and PDF URLs)

A design note worth recording: the final schema follows a **flat pattern** where each row in `documents` represents a single chunk, rather than a hierarchical pattern where one document contains many chunks. Osama set this up during Phase 1, and the ingestion pipeline was written to match. The two tables have some overlap (both store `chunk_text`), but queries are simpler and ingestion is straightforward.

### 2. RAG Ingestion Pipeline

Four Python modules under `backend/rag/`:

- `pdf_extractor.py` - extracts text from PDFs using pdfplumber, with cleanup for whitespace artifacts
- `text_chunker.py` - splits text into ~1000-character chunks with 200-character overlap, preferring natural breaks at paragraph and sentence boundaries
- `embedding_generator.py` - generates 1536-dim embeddings via OpenAI's `text-embedding-3-small` model, with batched API calls
- `ingest_knowledge_base.py` - orchestrates the full pipeline end to end

All four scripts share a common `__init__.py` that exports their public functions for use elsewhere in the backend.

### 4. Retrieval Layer

Two parts, both required:

**Postgres function (`match_knowledge_base`)** - runs inside Supabase, performs cosine similarity search using pgvector's `<=>` operator, joins `embeddings` back to `documents` for metadata, supports optional category filtering.

**Python wrapper (`backend/rag/retriever.py`)** - three public functions:
- `retrieve_context(query_text, match_count, category_filter, min_similarity)` - main retrieval function agents will use
- `retrieve_context_by_category(query_text, category, match_count)` - convenience wrapper for category-restricted searches
- `format_context_for_prompt(results)` - formats retrieval results as a clean string ready to inject into LLM prompts

The retriever includes a default similarity threshold of 0.3 to filter out weak matches. Queries that don't match anything meaningful in the knowledge base return an empty list rather than unrelated chunks.

## Issues Encountered and How They Were Handled

Documenting these explicitly because they shaped the final state of the knowledge base.

### PDF Extraction Quality

The Workitect Best Practices document (originally 120 chunks) extracted with severe text fusion issues. Words were concatenated without spaces across 77% of its chunks, making them unusable for semantic search. A diagnostic query measuring average space count per chunk revealed this clearly - Workitect had 14 spaces per 200 characters while healthy documents averaged 28 to 39.

**Resolution:** Replaced Workitect with the U.S. Department of Labor's "Competency Models: A Review of the Literature" (65 clean chunks). The DOL document is a stronger authoritative source and extracts cleanly because it's a native PDF rather than a browser-generated export.

Two First Round Review articles originally had the same issue because they had been saved via Print to PDF (which often produces image-based output). One was fixed by re-saving via File > Save As. The other remained broken after the second save attempt, likely because the article contained image-rendered text. That file was removed from the knowledge base.

### Script Import Error on First Run

The initial ingestion script tried to import from a non-existent `database` module, which caused a `ModuleNotFoundError`. The script was refactored to use the Supabase Python client directly, removing the unneeded module dependency. This also made the ingestion script self-contained and easier to reason about.

### Environment Variable Naming

The script originally expected `SUPABASE_KEY`, but the `.env` file used `SUPABASE_ANON_KEY` (Supabase's standard naming). The script was updated to match the existing convention rather than renaming the environment variable, since Osama's `supabase_client.py` already depended on the existing name.

### .env Location

The `.env` file was at the repository root (`M:\projects\Tarsheeh-cv\.env`) rather than inside `backend/`. Python's `load_dotenv()` searches relative to the script's location, so the backend couldn't find it. Resolved by moving the file to `backend/.env`.

## Known Limitations

Items that are intentionally not addressed in this phase but worth recording:

- **Document imbalance**: the OPM Structured Interview Guide has 98 chunks and tends to dominate results for interview-related queries. Not a real problem for hackathon use but would need reranking or per-document caps for production.
- **Two documents have minor extraction artifacts**: the CPHR Framework has 2 of 93 chunks with low space counts, and the OPM guide has 7 of 98. These are table rows and formatted headers rather than running prose. Not worth fixing.
- **No automated re-ingestion**: if someone adds a new PDF to `knowledge_base/`, they must clear the database and re-run the full ingestion. A smarter incremental ingest would skip already-ingested documents, but this isn't needed right now.

## Handoff for Phase 3

Everything agents need from Phase 2 is available via the RAG module. To use the retrieval function from an agent file:

```python
from rag.retriever import retrieve_context, format_context_for_prompt

results = retrieve_context(
    query_text="How should I score a candidate's communication skills?",
    match_count=5,
    category_filter="interview_scorecards"  # optional
)

context_string = format_context_for_prompt(results)
# inject context_string into your LLM prompt
```

The Interview Agent (my responsibility) will be the most active consumer of RAG context, using it to generate role-appropriate interview questions. The CV Analyzer (Osama's) will also benefit, since competency frameworks and resume evaluation guidelines both live in the knowledge base.
